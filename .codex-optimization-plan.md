# 给 Codex 的 Jindou Agent 架构重构方案

## 现状分析

当前对话流程：
1. 用户输入 → conversationOrchestrator.ts
2. → analyzeCreativeRequest（本地规则分析）
3. → buildAgentQuestions（本地硬编码问题，**完全不调用 LLM**）
4. → 用户点选项按钮
5. → 循环直到 questions 清空
6. → 最后才调 requestPromptFromLlm 生成提示词

**问题：DeepSeek 在对话过程中全程闲置，只有最后一步参与。**

## 目标架构

```
用户输入 → conversationOrchestrator
              │
              ├→ ① analyzeCreativeRequest（本地规则分析，给LLM参考）
              │
              └→ ② requestLlmChat（新接口）
                     │
                     ├→ systemPrompt（导演人设）
                     ├→ 全部历史消息
                     └→ 用户最新输入
                          │
                          ├→ LLM返回普通文本 → 直接展示给用户
                          └→ LLM返回JSON提示词 → PromptCard展示
```

## 需要改的文件

### 1. systemPrompt.ts — 重写导演人设（最重要）

**现状：** system prompt 写了很多硬编码的"反问规则"（视频类型不明确时问xxx，口播不明确时问xxx），把 LLM 框死了。

**改为：** 一个真正像导演的人设 prompt，让 LLM 自由发挥：

```
你叫 Jindou Agent，是一位有多年经验的 AI 视频广告导演。

你的工作方式：
- 用户跟你聊天，你像一位导演一样引导他
- 用户说不清楚时，你主动推断和补全，而不是把所有问题丢回去
- 用户说"高级一点"，你自动理解成"电影质感、柔和侧光、慢节奏"，而不是追问"哪种高级"
- 用户说"护肤品广告"，你自动推断场景（浴室/卧室）、动作（涂抹/按压）、情绪（放松/焕亮）
- 非关键问题（口播、镜头形式、参考图）你帮用户做决策，只把真正需要用户拍板的问题抛出来
- 你随时可以输出提示词，不需要等用户说"可以了"

你的输出有两种模式：
1. **聊天模式**：跟用户自然对话，展示你的理解，引导用户完善想法
2. **生成模式**：当信息足够时，直接输出 JSON 格式的提示词

生成模式的 JSON 格式：
{"summary": "一句话总结", "segments": [{"title": "提示词 1", "timeRange": "0-15 秒", "duration": 15, "videoType": "...", "model": "Seedance", "shotMode": "multi", "prompt": "完整提示词正文..."}]}

生成规则：
- 每条提示词最多15秒，超过自动拆分
- 拆分后的每条提示词必须独立可生成
- 禁止写"与上一段一致""延续前一段""后期加入""可根据需要""参考一下"等模糊表述
- 提示词正文只写画面描述和指令，不写解释性语言
- 提示词正文不包含模型名称
```

### 2. conversationOrchestrator.ts — 让LLM主导对话

**现状：**
```typescript
const questions = buildAgentQuestions(analysis);
if (questions.length > 0) {
  return { messages: [本地硬编码问题] };  // ❌
}
const llmResult = await requestPromptFromLlm({...});
```

**改为：**
```typescript
// 1. 本地规则分析（给LLM参考信息）
const analysis = analyzeCreativeRequest(brief, ...);

// 2. 构建给LLM的消息列表
const llmMessages = buildLlmMessages({
  systemPrompt: JINDOU_SYSTEM_PROMPT,
  history: 历史对话,
  userInput: 用户最新输入,
  analysisSummary: 简化的分析结果（让LLM参考但不限制它）
});

// 3. 调LLM
const llmReply = await requestLlmChat({ messages: llmMessages });

// 4. 解析回复
if (llmReply.isJson && llmReply.segments) {
  // LLM生成了提示词 → 质量守卫检查 → 展示卡片
  const safePrompts = llmReply.segments.map(s => ({
    ...s,
    prompt: guardPromptText(s.prompt).cleanPrompt
  }));
  return { messages: [createMessage({ kind: "prompts", prompts: safePrompts })] };
} else {
  // LLM在聊天 → 直接展示对话文本
  return { messages: [createMessage({ kind: "text", content: llmReply.text })] };
}
```

### 3. llmClient.ts — 新增通用聊天接口

**现状：** 只有 `requestPromptFromLlm`（专门生成提示词的高层接口）。

**改为：** 
- 新增 `requestLlmChat(messages)` — 通用聊天接口，直接调 API
- 保留 `requestPromptFromLlm` — 可以去掉或保留作为备用
- `callChatCompletion` — 保留，是底层 HTTP 调用

新接口逻辑：
```typescript
async function requestLlmChat(args: {
  messages: Array<{role, content}>,
  temperature?: number
}): Promise<{
  text: string,
  isJson: boolean,
  segments?: LlmPromptResponse['segments']
}> {
  const response = await callChatCompletion(args.messages, args.temperature);
  if (!response) return { text: "抱歉，我现在暂时无法回应，请稍后再试。", isJson: false };
  
  // 尝试解析JSON
  const parsed = tryParseJson(response);
  if (parsed?.segments?.length) {
    return { text: parsed.summary || "", isJson: true, segments: parsed.segments };
  }
  
  return { text: response, isJson: false };
}
```

### 4. App.tsx — 前端兼容

**现状：** `handleSend` 调 `runAgentTurn`，只处理 `questions` 和 `prompts` 两种 message kind。

**改为：**
- 新增 `text` 类型的 message kind（已在 types/chat.ts 中定义了但不常用）
- `MessageBubble.tsx` 中 `kind === "text"` 时正常显示对话文本
- 这样 LLM 返回的自然对话文本就能直接展示了

## 实施顺序

1. **systemPrompt.ts** → 先重写人设（最优先）
2. **llmClient.ts** → 新增 requestLlmChat
3. **conversationOrchestrator.ts** → 让LLM主导
4. **App.tsx** → 小改前端兼容

## 注意事项

- LLM API 可能失败 → 保留本地 fallback（硬编码问题 + 本地提示词模板）
- promptQualityGuard.ts 的质量检查仍然要跑 → 安全网
- system prompt 不要写太多限制，给 LLM 发挥空间
- 新老架构可以并行：先让LLM主导对话，本地规则作为兜底
